# AI 閱讀入口

本文件是給 AI、agent 或新接手開發者的最短閱讀路徑。目標是先建立全局理解，再只讀必要文件，避免一次讀完 12 套方案造成上下文過長與風格混用。

## 先讀結論

這個倉庫不是實際 Web App，而是 WebUI 方向資料庫。每次實作前只能選一個唯一主方案，再用共用規格與小組件文件補齊細節。

AI 產出時必須遵守：

1. 先確認唯一主方案。
2. 先輸出資訊架構、design token、元件清單、狀態、RWD、可及性。
3. 最後才輸出程式碼或完整文件。
4. 不混用其他方案的主色、圓角、陰影、資訊密度與互動語言。
5. 不使用 emoji 當正式 UI icon。

## 最短閱讀路徑

### 只要快速了解專案

依序讀：

1. `README.md`
2. `共用規格/方案選擇矩陣.md`
3. `共用規格/AI輸出契約.md`
4. `小組件/README.md`

不要先讀所有 `方案-*` 資料夾。

### 要實作某個頁面

依序讀：

1. `README.md`
2. `共用規格/方案選擇矩陣.md`
3. `共用規格/AI實作交接卡.md`
4. 選定主方案的 `README.md`
5. 選定主方案的 `design-system.md`
6. 選定主方案的 `components.md`
7. 選定主方案的 `pages.md`
8. `小組件/README.md`
9. 只讀本次會用到的小組件文件
10. `共用規格/AI輸出契約.md`

### 要新增或大改方案

依序讀：

1. `README.md`
2. `共用規格/細化檢查清單.md`
3. `共用規格/方案品質門檻.md`
4. 相近的 1-2 個既有方案，不要超過 2 個
5. `小組件/README.md`
6. `agent/項目表.md`
7. `agent/memory.md`

### 要審查 PR

依序讀：

1. `共用規格/方案品質門檻.md`
2. `共用規格/細化檢查清單.md`
3. 被修改方案的 4 份文件
4. `README.md` 是否同步更新
5. `agent/項目表.md`、`agent/memory.md` 是否同步更新

## AI 閱讀規則

- 先讀共用規格，再讀單一方案。
- 只讀與任務直接相關的方案，不要把 12 套方案全部塞入上下文。
- 若使用者沒有指定方案，先用 `方案選擇矩陣.md` 推薦 1 套，不要自行混合多套。
- 若需要小組件，先從 `小組件/` 取共用結構，再套用主方案的 token。
- 若共用小組件與主方案文件衝突，以主方案 `design-system.md` 與 `components.md` 為準。
- 若使用者要求「漂亮、現代、科技感」但沒有產品脈絡，要先要求或補齊交接卡欄位。

## 低上下文模式

如果可用上下文很少，只讀這 5 個檔案：

1. `README.md`
2. `共用規格/方案選擇矩陣.md`
3. `共用規格/AI實作交接卡.md`
4. 選定方案的 `README.md`
5. `共用規格/AI輸出契約.md`

小組件只在需要具體元件時再讀。

## 任務判斷表

| 使用者需求 | 先讀 | 接著讀 | 產出前檢查 |
| --- | --- | --- | --- |
| 做後台頁面 | `方案選擇矩陣.md` | 方案 01 或 05 的方案文件 | `AI輸出契約.md` |
| 做 AI 工具 | `方案選擇矩陣.md` | 方案 02 的方案文件 | `小組件/資料與狀態元件.md` |
| 做表單或設定 | `小組件/基礎元件.md` | 主方案 `components.md` | 表單狀態與 validation |
| 做列表或資料頁 | `小組件/資料與狀態元件.md` | 主方案 `pages.md` | loading、empty、error |
| 做彈窗或導覽 | `小組件/導覽與浮層元件.md` | 主方案 `components.md` | focus、keyboard、關閉規則 |
| 新增方案 | `細化檢查清單.md` | `方案品質門檻.md` | 是否達最低門檻 |

## AI 回答前自檢

```text
我是否知道唯一主方案？
我是否只讀了必要文件？
我是否知道哪些小組件會用到？
我是否先套用主方案 token，再套用小組件結構？
我是否列出 loading、empty、error、disabled、success？
我是否避免 emoji icon 與跨方案混風格？
```
