# S12RYT WebUI 方向資料庫

本倉庫用於沉澱 WebUI 的設計方向、元件風格、頁面結構與 AI 實作規則。目標是讓後續 AI 或開發者在新增頁面時能先選定主方案，再依同一套規格延伸，避免每次產出風格不統一。

## 使用方式

1. 若由 AI、agent 或新接手開發者閱讀，先看 `AI閱讀入口.md`，依任務只載入必要文件；人類快速瀏覽可從本文件開始。
2. 到 `共用規格/` 讀取跨方案規則：
   - `方案選擇矩陣.md`：用產品類型、資訊密度、信任感、視覺強度快速選方案。
   - `細化檢查清單.md`：實作前、中、後檢查 layout、token、元件狀態、RWD、a11y。
   - `AI輸出契約.md`：給 AI 的固定輸入/輸出格式，避免風格漂移。
   - `方案品質門檻.md`：用共同評分表檢查每套方案是否足夠落地。
   - `AI實作交接卡.md`：把主方案、頁面範圍、狀態、RWD、禁用風格整理成可交給 AI 的交接模板。
3. 依產品需求從根目錄的方案資料夾中挑選一個唯一主方案。
4. 實作前先填寫 `共用規格/AI實作交接卡.md`，避免只給「做一個漂亮頁面」這類模糊指令。
5. 進入選定方案資料夾，依序閱讀：
   - `README.md`：定位、適用/不適用、使用者與實作優先順序。
   - `design-system.md`：色彩、字體、間距、layout、狀態、動效與禁用規則。
   - `components.md`：基礎元件、情境元件、狀態規則與內容寫法。
   - `pages.md`：頁面模板、資訊架構、主要流程與 AI 提示模板。
6. 需要具體元件時，讀 `小組件/README.md` 與對應小組件文件；小組件只提供結構、狀態與互動規則，視覺 token 仍以唯一主方案為準。
7. 若需求需要混合多種方向，必須先指定主方案，只借用其他方案的單一元素，且不能混用色彩、圓角、陰影、排版密度。
8. 新增或大幅修改方案時，使用 `共用規格/方案品質門檻.md` 做 PR review；未達最低合格門檻時不要直接交給 AI 實作。

## 共通 WebUI 原則

- 以桌面與平板體驗為主，手機版至少確保可閱讀、可操作、無水平捲動。
- 色彩、字體、圓角、陰影、間距需集中定義，不要每個頁面臨時決定。
- 所有可互動元素需有 hover、focus、active、disabled、loading 狀態。
- 不使用 emoji 當正式 UI icon；使用一致的 SVG icon 系統。
- 文字階層需明確：頁面標題、區塊標題、內文、輔助文字、錯誤文字不能混用。
- 重要操作要有確認、載入中、成功、失敗、復原或重新嘗試狀態。
- 表格、卡片、表單、空狀態、錯誤狀態、權限不足狀態都要有可複用模式。
- 每個頁面要先定義資訊密度，不要在同一產品中混用高密度後台與大留白 landing 的節奏。
- 每個方案都要明確指定「不要做什麼」，避免 AI 自行加入不合適的玻璃擬態、霓虹、漸層或過度動畫。

## 建議預設技術方向

若尚未決定框架，建議先用下列抽象規則描述 UI：

- Framework：React / Next.js / Vue 其一。
- Styling：Tailwind CSS 或同等 utility-first design token。
- Icons：Lucide / Heroicons 擇一，不混用多套 icon。
- Charts：Recharts / ECharts 擇一，依資料量與互動需求決定。
- Components：先參考 `小組件/` 的拆分規則，再建立 Button、Input、Card、Dialog、Tabs、Table、Toast、EmptyState、Skeleton、Tooltip。
- Theme：把方案的 CSS token 映射為 `bg`、`surface`、`text`、`muted`、`primary`、`success`、`warning`、`danger`。

## 方案選擇速查

| 方案 | 適合場景 | 第一印象 | 主要風格 | 資訊密度 |
| --- | --- | --- | --- | --- |
| 方案 01 | B2B 後台、管理系統、SaaS | 穩定、可信、專業 | 淺色、藍灰、資料密度中高 | 中高 |
| 方案 02 | AI 工具、開發者平台、模型工作台 | 高科技、專注、沉浸 | 暗色、霓虹點綴、命令介面感 | 中 |
| 方案 03 | 小工具、設定面板、效率產品 | 清楚、快速、無負擔 | 極簡、白底、高留白 | 低中 |
| 方案 04 | 官網、產品介紹、轉換頁 | 有品牌感、易理解 | 大視覺、漸層、敘事分區 | 低 |
| 方案 05 | 監控中心、分析平台、營運儀表板 | 即時、精準、掌控感 | 圖表優先、狀態色明確 | 高 |
| 方案 06 | FinTech、錢包、投資、支付、帳務 | 安全、精準、可信任 | 高信任金融科技、低噪音視覺 | 中高 |
| 方案 07 | 電商、訂單、商品、購物流程 | 清楚、轉換、可操作 | 商品卡、篩選、購物車、營運數據 | 中 |
| 方案 08 | 社群、論壇、內容平台、會員互動 | 活躍、可讀、可參與 | Feed、討論串、作者資訊、互動狀態 | 中 |
| 方案 09 | 文件、知識庫、Help Center、開發者文件 | 可搜尋、可掃讀、可信 | 側欄目錄、內容排版、版本提示 | 中 |
| 方案 10 | 作品集、媒體展示、創作者首頁 | 風格、展示、沉浸 | 大圖、媒體卡、案例敘事、品牌個性 | 低中 |
| 方案 11 | 線上課程、學習平台、訓練系統 | 有進度、有陪伴、可完成 | 課程章節、進度、測驗、筆記 | 中 |
| 方案 12 | 活動頁、遊戲化任務、互動體驗 | 有趣、刺激、引導行動 | 任務卡、進度、獎勵、即時回饋 | 中 |

## 選擇建議

- 不確定要做一般後台：先選 `方案-01`。
- 產品核心是 AI 對話、模型、agent、prompt：選 `方案-02`。
- 只有單一任務、單一輸入輸出：選 `方案-03`。
- 目標是轉換、介紹、等待名單：選 `方案-04`。
- 主要是大量圖表、告警、即時數據：選 `方案-05`。
- 牽涉金流、帳務、投資、風險提示：選 `方案-06`。
- 牽涉商品列表、購物車、訂單流程：選 `方案-07`。
- 牽涉貼文、留言、通知、會員互動：選 `方案-08`。
- 牽涉長文文件、API 文件、FAQ、站內搜尋：選 `方案-09`。
- 牽涉視覺展示、攝影、案例、個人品牌：選 `方案-10`。
- 牽涉課程、章節、學習進度、測驗：選 `方案-11`。
- 牽涉任務、徽章、排行榜、活動轉換：選 `方案-12`。

## AI 實作提示

每次讓 AI 新增 WebUI 時，請附上：

```text
請使用「方案-XX」作為唯一主視覺方向，遵守該資料夾內 README.md、design-system.md、components.md、pages.md，以及共用規格/AI輸出契約.md。
請先閱讀 AI閱讀入口.md，並只載入本次任務需要的方案與小組件文件。
請先填寫或參考共用規格/AI實作交接卡.md，明確說明頁面範圍、主要任務、資料密度、必備狀態與禁用風格。
若使用小組件，請列出小組件來源文件；小組件只提供結構與互動，所有色彩、圓角、陰影、字級與間距都必須套用主方案 design-system.md。
請先輸出本頁資訊架構、使用的 design token、需要的元件與狀態，再輸出程式碼。
不要混用其他方案的色彩、圓角、陰影與排版密度，除非我明確指定。
不要使用 emoji 當正式 UI icon。
```
