# AI 輸出契約

本文件定義交給 AI 實作 WebUI 時的固定契約。每次生成頁面、元件或方案，都必須先收斂規格，再輸出結果。

## 使用者輸入模板

```text
主方案：方案-XX-名稱
頁面/元件名稱：
產品背景：
目標使用者：
主要任務：
必備內容：
資料密度：低 / 中 / 中高 / 高
互動需求：
RWD：桌面優先 / 手機優先 / 皆需完整
禁用風格：
技術限制：
小組件來源：使用 `小組件/` 的哪些文件與哪些元件；若不用請寫無
```

## AI 回答必須包含

1. `方案確認`：說明採用哪個主方案，以及不會混用哪些風格。
2. `資訊架構`：列出頁面區塊順序與每區塊用途。
3. `Design token 使用`：列出背景、表面、文字、主色、狀態色、圓角、陰影。
4. `元件清單`：列出使用的基礎元件、情境元件與引用的 `小組件/` 文件。
5. `小組件套用方式`：若使用小組件，說明哪些規則來自小組件、哪些視覺 token 回到主方案。
6. `狀態設計`：至少包含 loading、empty、error、disabled、success。
7. `RWD 策略`：桌面、平板、手機如何變化。
8. `可及性`：focus、label、對比、鍵盤操作、替代文字。
9. `實作內容`：最後才輸出程式碼或文件。

## 禁止輸出模式

- 不可只給一段程式碼，沒有說明依據哪個方案。
- 不可自行加入 emoji icon。
- 不可用「看起來現代」這種模糊描述取代 token。
- 不可在同一頁混用多套主色、圓角、陰影與排版密度。
- 不可把 `小組件/` 當成另一套視覺方案；小組件只提供結構、狀態與互動，視覺 token 必須回到主方案。
- 不可只處理 happy path，忽略空資料、錯誤、載入、權限不足。
- 不可讓 hover 使用會推擠版面的 transform 或 border 寬度變化。

## 固定回答骨架

```text
## 方案確認

## 資訊架構

## Design token 使用

## 元件清單

## 小組件套用方式

## 狀態設計

## RWD 策略

## 可及性檢查

## 實作內容
```

## 元件輸出規則

建立元件時必須定義：

- Props 或可配置欄位。
- variants 與 sizes。
- disabled、loading、error 狀態。
- keyboard/focus 行為。
- 使用情境與不適合情境。

## 頁面輸出規則

建立頁面時必須定義：

- Page header：標題、說明、主要 CTA。
- Main content：核心任務區，不超過 3 個主區塊。
- Secondary content：次要資訊，不能搶主任務注意力。
- Feedback：toast、inline alert、skeleton、empty state。
- Navigation：breadcrumb、tabs、side nav 或底部導覽。

## 審查提示

完成後請自查：

```text
我是否明確使用單一主方案？
我是否把每個顏色與間距都對應到 design token？
我是否列出使用的小組件來源，並說明如何套回主方案 token？
我是否定義了 loading、empty、error、disabled？
我是否避免 emoji icon 與混合風格？
我是否能讓下一個 AI 只靠這份輸出接手？
```
