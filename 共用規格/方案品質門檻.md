# 方案品質門檻

本文件用於評估每一套 WebUI 方案是否已經足夠讓 AI 或開發者穩定接手。`細化檢查清單.md` 偏向逐項檢查，本文件偏向交付前的品質判斷與最低門檻。

## 使用時機

- 新增一套方案後，確認是否能放入主資料庫。
- 修改既有方案後，確認沒有讓風格變模糊。
- 實作頁面前，確認選定方案是否足夠支撐該產品。
- PR review 時，用一致標準判斷文件是否可合併。

## 最低合格門檻

一套方案至少必須同時通過以下條件：

1. `README.md` 能說清楚定位、典型使用者、適合/不適合場景、資訊密度與風格邊界。
2. `design-system.md` 至少有色彩、字體、字級、間距、圓角、陰影、layout、狀態、動效與禁用規則。
3. `components.md` 至少有 8 個可落地元件，且包含狀態、互動與內容規則。
4. `pages.md` 至少有 5 個核心頁型，且每個頁型都有資訊順序與狀態要求。
5. 至少定義 loading、empty、error、disabled、success、permission denied 或等價狀態。
6. 必須明確寫出「不要做什麼」，避免 AI 自行套用不相容風格。
7. 必須能用一句話回答：這套方案與其他方案最大的差異是什麼。

未達以上任一條，不建議作為主方案使用。

## 評分表

每項 0-2 分，滿分 20 分。

| 評估項目 | 0 分 | 1 分 | 2 分 |
| --- | --- | --- | --- |
| 方案定位 | 只描述視覺 | 有產品類型但不夠明確 | 明確描述產品、使用者、任務與情緒 |
| 風格邊界 | 沒有禁用規則 | 有零散禁用描述 | 清楚說明可做與不可做 |
| Design token | 只有顏色或模糊形容 | 有部分 token | 色彩、字體、間距、圓角、陰影、狀態完整 |
| Layout | 只列區塊 | 有桌面結構 | 桌面、平板、手機與密度規則完整 |
| 元件完整度 | 少於 5 個元件 | 有常見元件 | 有基礎元件與情境專用元件 |
| 頁型完整度 | 少於 3 個頁型 | 有常見頁型 | 覆蓋 dashboard/list/detail/form 或該方案核心流程 |
| 狀態設計 | 只處理正常狀態 | 有部分狀態 | loading、empty、error、disabled、success、permission 完整 |
| 可及性 | 未提及 | 只提對比或 focus | label、focus、keyboard、contrast、alt、非顏色狀態完整 |
| AI 可執行性 | 讀完仍需大量猜測 | 可做但容易漂移 | 有提示模板、輸出契約、元件與狀態約束 |
| 可維護性 | 方案容易互相混用 | 有基本分類 | 有命名、混用規則、審查門檻與交接模板 |

## 評級

| 分數 | 評級 | 判斷 |
| --- | --- | --- |
| 18-20 | A | 可直接作為主方案並進入實作。 |
| 15-17 | B | 可使用，但 PR 前應補弱項。 |
| 12-14 | C | 可作草稿，不建議直接交給 AI 實作。 |
| 0-11 | D | 需要重寫或大幅補強。 |

## PR Review 必問問題

1. 這次新增或修改是否讓某一套方案更清楚，而不是更模糊？
2. 有沒有新增會導致跨方案混色、混圓角、混陰影、混密度的描述？
3. 元件規則是否包含 disabled、loading、empty、error 或等價狀態？
4. 頁型是否有明確資訊順序，而不是只列元件名稱？
5. 手機 375px、平板 768px、桌面 1024px 以上是否都有合理策略？
6. 是否避免 emoji icon、模糊的「現代感」、不受控玻璃擬態與過度動畫？
7. 下一個 AI 是否能只靠本方案與共用規格產出一致 UI？

## 使用範例

```text
方案：方案-07-電商營運購物體驗
評分：17 / 20，B
弱項：可及性只提到部分互動狀態，需補鍵盤操作與替代文字。
結論：可以作為主方案，但在進入實作前需補產品圖片 alt、篩選器鍵盤操作與 checkout 表單 label。
```

## 不合格時的修正順序

1. 先補 `README.md` 的定位與風格邊界。
2. 再補 `design-system.md` 的 token 與 layout。
3. 接著補 `components.md` 的狀態與互動。
4. 最後補 `pages.md` 的頁型流程與 AI 提示模板。

不要反過來先寫漂亮頁面範例，否則 AI 會複製局部視覺而忽略整體系統。
