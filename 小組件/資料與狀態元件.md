# 資料與狀態元件

這一組元件處理高密度資料、載入回饋與資料缺漏情境。它們通常出現在後台、監控、電商、文件中心與 AI 工作台。

## Table

### 用途

呈現可排序、可篩選、可批次操作的結構化資料。

### 常見 variants

- `default`：標準資料表。
- `compact`：高密度後台。
- `striped`：視覺上更容易掃讀。
- `interactive`：可點擊列、行內操作與篩選。

### 必備狀態

- loading
- empty
- error
- filtered-empty
- selected rows
- disabled rows

### 規則

- Loading 時優先顯示 skeleton，而不是空白表格。
- 空資料要告訴使用者下一步，不要只顯示「No data」。
- 行內操作最多保留 2-3 個高頻動作，其餘收進更多選單。
- 手機上表格若無法維持可讀性，應改成卡片列表或 detail drawer。

## List / Card List

### 用途

用於內容摘要、任務列表、通知、搜尋結果、文章、商品與文件。

### 規則

- 每列應有一個明確主資訊與一個次資訊。
- 行為型列表要支援選取、排序或篩選。
- 長文字應截斷並提供展開或詳情入口。

## MetricCard

### 用途

呈現 KPI、金額、數量、成長率、狀態摘要與比較值。

### 必備內容

- title
- value
- delta / trend
- helper context
- optional mini sparkline

### 規則

- 數值變化需顯示上升、下降或持平語意，不可只靠顏色。
- 金額與百分比格式要一致。
- 若指標不可靠或尚未更新，要明確標示 stale 或 delayed。

## ChartPanel

### 用途

包裝圖表與其說明、篩選條件、圖例與時間範圍。

### 規則

- 圖表旁邊要有能讀懂數字的文字摘要。
- 沒有資料時不要保留空圖框，應顯示空狀態與解釋。
- 需要 tooltip、legend、縮放或切換 series 時，必須定義互動順序。

## EmptyState

### 用途

資料不存在、篩選後無結果、尚未建立內容、權限不足或初始空白。

### 必備組成

- title
- description
- optional illustration or icon
- primary action
- secondary action

### 規則

- 空狀態要能區分「還沒開始」與「真的沒有資料」。
- 可行時要提供下一步，例如新增、重試、清除篩選、聯絡管理員。
- 不可只有大圖與口號，沒有實際操作。

## Skeleton

### 用途

載入中時提供結構提示，避免畫面跳動。

### 規則

- Skeleton 形狀要貼近真實內容，不要亂畫。
- 載入時間短時可用輕量 placeholder，不必過度動畫。
- 不同區塊的 skeleton 速度與密度應與主方案一致。

## ErrorState

### 用途

資料讀取失敗、儲存失敗、權限不足、服務中斷、驗證失敗。

### 必備組成

- error title
- reason or hint
- retry action
- alternative action or contact path

### 規則

- 錯誤資訊要可執行，不只是情緒化文案。
- 若是局部失敗，要避免整頁報錯。
- 重新嘗試與回報問題需清楚分開。

## PermissionNotice

### 用途

說明唯讀、權限不足、禁止操作、需升級方案或需登入。

### 規則

- 要說明誰可以做、要怎麼取得權限。
- 不可只把按鈕 disabled 而不解釋原因。
- 重要情境下要提供申請權限、聯絡管理員或登入入口。

## Toast / InlineAlert

### 用途

呈現成功、警告、失敗、資訊與提示性訊息。

### 規則

- Success toast 不可過度打擾，Error inline alert 要保留上下文。
- Toast 與 inline alert 的訊息長度應短且明確。
- 錯誤類提示不能只用顏色，需有文字與圖示語意。
