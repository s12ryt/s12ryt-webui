# 方案 09：知識庫文件中心

## 定位

適合產品文件、開發者文件、Help Center、FAQ、內部知識庫、API 文件、教學文章、版本公告與搜尋導向內容平台。

## 設計關鍵字

- 可搜尋
- 可掃讀
- 可信
- 長文友善
- 結構清楚
- 版本可追溯

## 視覺摘要

以閱讀體驗與資訊架構為核心。畫面需要有清楚側欄、文章內容、目錄、搜尋與版本提示。視覺不追求炫技，而是讓使用者快速找到答案並判斷內容是否適用於目前版本。

## 典型使用者

- 查找產品操作方式的一般使用者。
- 查 API、SDK、CLI 指令的開發者。
- 撰寫與維護文件的內容管理者。
- 需要搜尋內部規範與 SOP 的團隊成員。

## 適合頁面

- 文件首頁
- 文章閱讀頁
- API Reference
- 搜尋結果頁
- FAQ 頁
- 版本公告 / Changelog
- 文件編輯或審核後台

## 不適合

- 高轉換 landing 作為唯一目的的頁面。
- 商品購物流程。
- 即時圖表監控。

## 資訊密度

中。導航與搜尋可較密，文章內容需保持良好行高與最大寬度；API reference 可提高密度但要有清楚錨點與程式碼區塊。

## 實作優先順序

1. 建立文件 shell：頂部搜尋、左側目錄、文章內容、右側 TOC。
2. 建立文章排版：標題、段落、列表、callout、code block、table。
3. 建立搜尋結果與空結果。
4. 補版本、權限、編輯、回饋與多語系狀態。

## 風格邊界

- 可以有品牌色，但文章閱讀區需克制。
- 程式碼區塊可以暗色，但不要讓整頁變成 AI 工作台。
- 文件頁重點是「找到答案」，不是展示視覺效果。
